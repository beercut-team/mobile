# Сценарий использования мобильного приложения

## Обзор системы

Мобильное приложение для управления пациентами офтальмологической клиники с поддержкой iOS, Android и Web. Система обеспечивает взаимодействие между участковыми врачами, хирургами, пациентами и администраторами.

## Роли пользователей

### 1. Участковый врач (DISTRICT_DOCTOR)
- Создает карточки пациентов
- Заполняет паспортные данные и диагнозы
- Проводит IOL-расчеты
- Добавляет комментарии и документы
- Отправляет пациентов на проверку хирургу

### 2. Хирург (SURGEON)
- Просматривает пациентов на проверке
- Проверяет чеклисты предоперационной подготовки
- Одобряет или отклоняет пациентов
- Назначает даты операций
- Отслеживает статус операций

### 3. Пациент (PATIENT)
- Входит по коду доступа
- Просматривает свой статус
- Видит дату операции
- Получает уведомления об изменениях

### 4. Администратор (ADMIN)
- Управляет пользователями
- Модерирует контент
- Просматривает статистику
- Управляет районами и настройками

## Основные сценарии использования

### Сценарий 1: Регистрация нового пациента

**Участники:** Участковый врач

**Шаги:**
1. Врач открывает приложение и авторизуется
2. Переходит на вкладку "Пациенты"
3. Нажимает кнопку "+" для создания нового пациента
4. Заполняет форму:
   - ФИО пациента
   - Дата рождения
   - Телефон и адрес
   - СНИЛС, паспорт, полис ОМС
   - Диагноз (например, "Катаракта правого глаза")
   - Тип операции (Факоэмульсификация)
   - Глаз (OD - правый, OS - левый, OU - оба)
   - Район
5. Сохраняет карточку пациента
6. Система генерирует уникальный код доступа для пациента
7. Врач сообщает код пациенту для отслеживания статуса

**Результат:** Пациент создан со статусом "NEW"

---

### Сценарий 2: Подготовка пациента к операции

**Участники:** Участковый врач

**Шаги:**
1. Врач открывает карточку пациента
2. Переходит на вкладку "Документы"
3. Загружает необходимые файлы:
   - Результаты анализов
   - Снимки глазного дна
   - Заключения специалистов
4. Переходит на вкладку "IOL Калькулятор"
5. Вводит данные биометрии:
   - Аксиальная длина глаза (23.5 мм)
   - Кератометрия K1 и K2 (43.5 / 44.0 D)
   - Глубина передней камеры (3.2 мм)
   - Целевая рефракция (-0.5 D)
6. Выбирает формулу расчета (SRK/T, Haigis, Hoffer Q)
7. Система рассчитывает силу ИОЛ
8. Врач сохраняет расчет
9. Переходит на вкладку "Чеклист"
10. Отмечает выполненные пункты подготовки:
    - ✅ Общий анализ крови
    - ✅ Биохимия крови
    - ✅ ЭКГ
    - ✅ Флюорография
    - ⏳ Консультация терапевта (в процессе)
11. Добавляет комментарий: "Пациент готов к операции. Все анализы в норме."
12. Изменяет статус на "PREPARATION"
13. После завершения подготовки меняет статус на "REVIEW_NEEDED"

**Результат:** Пациент отправлен на проверку хирургу

---

### Сценарий 3: Проверка и одобрение хирургом

**Участники:** Хирург

**Шаги:**
1. Хирург открывает приложение
2. Видит уведомление о новом пациенте на проверке
3. Переходит на вкладку "Пациенты"
4. Фильтрует список по статусу "На проверке"
5. Открывает карточку пациента "Айаана Николаева"
6. Просматривает вкладки:
   - **Паспортные данные**: проверяет корректность ФИО, даты рождения
   - **Документы**: просматривает загруженные анализы и снимки
   - **IOL Калькулятор**: проверяет расчеты силы линзы
   - **Чеклист**: видит прогресс 14/15 (93% выполнено)
   - **Комментарии**: читает заметки врача
7. Если все в порядке:
   - Нажимает кнопку "Одобрить"
   - Добавляет комментарий: "Пациент одобрен для операции"
   - Статус меняется на "APPROVED"
8. Если есть замечания:
   - Нажимает кнопку "Отклонить"
   - Указывает причину: "Требуется дополнительное обследование сетчатки"
   - Отмечает комментарий как срочный (🔴)
   - Статус меняется на "REJECTED"

**Результат:** Пациент одобрен или отклонен с комментариями

---

### Сценарий 4: Назначение операции

**Участники:** Хирург

**Шаги:**
1. Хирург открывает список одобренных пациентов
2. Выбирает пациента "Сардаана Васильева"
3. Нажимает кнопку "Назначить операцию"
4. Выбирает дату и время операции в календаре
5. Подтверждает назначение
6. Статус меняется на "SURGERY_SCHEDULED"
7. Система отправляет push-уведомление пациенту
8. Хирург может скачать PDF-документы:
   - Маршрутный лист пациента
   - Отчет по чеклисту

**Результат:** Операция назначена, пациент уведомлен

---

### Сценарий 5: Отслеживание статуса пациентом

**Участники:** Пациент

**Шаги:**
1. Пациент открывает приложение
2. Выбирает "Вход по коду доступа"
3. Вводит код, полученный от врача (например, "88437f5b")
4. Видит свою карточку с информацией:
   - ФИО
   - Текущий статус: "Операция назначена"
   - Дата операции: "15 марта 2026, 10:00"
5. Получает push-уведомления при изменении статуса:
   - "Ваша заявка одобрена хирургом"
   - "Операция назначена на 15 марта"
   - "Напоминание: операция завтра в 10:00"

**Результат:** Пациент информирован о своем статусе

---

### Сценарий 6: Работа с комментариями

**Участники:** Врач, Хирург

**Шаги:**
1. Врач добавляет комментарий к пациенту:
   - "Пациент принимает антикоагулянты, требуется консультация терапевта"
   - Отмечает как срочный (🔴)
2. Хирург получает уведомление о срочном комментарии
3. Открывает карточку пациента
4. Читает комментарий
5. Отвечает: "Получена консультация терапевта. Можно продолжать подготовку."
6. Комментарий отмечается как прочитанный

**Результат:** Обеспечена коммуникация между врачом и хирургом

---

### Сценарий 7: Офлайн-режим

**Участники:** Участковый врач

**Шаги:**
1. Врач находится в районе без интернета
2. Открывает приложение (работает офлайн)
3. Просматривает ранее загруженные карточки пациентов
4. Вносит изменения:
   - Обновляет комментарий
   - Отмечает пункты чеклиста
5. Изменения сохраняются локально
6. При появлении интернета:
   - Приложение автоматически синхронизирует данные
   - Показывает индикатор синхронизации
   - Уведомляет об успешной синхронизации

**Результат:** Работа продолжается без интернета

---

### Сценарий 8: Модерация контента

**Участники:** Администратор

**Шаги:**
1. Администратор открывает вкладку "Модерация"
2. Видит список пациентов, требующих проверки
3. Просматривает загруженные документы и комментарии
4. Удаляет некорректный контент при необходимости
5. Блокирует пользователей при нарушениях
6. Просматривает статистику:
   - Количество пациентов по статусам
   - Активность пользователей
   - Количество операций за период

**Результат:** Контент модерирован, статистика обновлена

## Типы уведомлений

Система поддерживает 5 типов push-уведомлений:

1. **STATUS_CHANGE** (🔄) - Изменение статуса пациента
   - "Пациент Айаана Николаева: статус изменен на 'Одобрен'"

2. **NEW_COMMENT** (💬) - Новый комментарий
   - "Новый комментарий к пациенту Ньургун Иванов"

3. **SURGERY_SCHEDULED** (📅) - Операция назначена
   - "Операция назначена на 15 марта 2026, 10:00"

4. **CHECKLIST_EXPIRY** (⚠️) - Истечение срока чеклиста
   - "Срок действия чеклиста истекает через 3 дня"

5. **SURGERY_REMINDER** (🔔) - Напоминание об операции
   - "Напоминание: операция завтра в 10:00"

## Особенности интерфейса

### Адаптивность
- Поддержка светлой и темной темы
- Режим высокой контрастности для слабовидящих
- Увеличение шрифта (2x) в режиме доступности

### Навигация
- Нижняя панель с вкладками (Главная, Пациенты, Уведомления, Профиль)
- Плавающая кнопка "+" для быстрого создания пациента
- Свайп для обновления списков (pull-to-refresh)

### Платформенные особенности
- iOS: SF Symbols для иконок, тактильная обратная связь (Haptics)
- Android: Material Icons, стандартные паттерны навигации
- Web: Адаптивная верстка, поддержка клавиатуры

## Технические возможности

### Интеграция с медицинскими стандартами
- **ICD-10**: Коды диагнозов (H25.1 - Возрастная катаракта)
- **SNOMED CT**: Коды процедур (397544007 - Факоэмульсификация)
- **LOINC**: Коды наблюдений (79893-4 - Острота зрения)
- **HL7 FHIR R4**: Экспорт данных в формате FHIR Bundle

### Интеграция с внешними системами
- **ЕМИАС** (Москва): Экспорт пациентов в единую медицинскую систему
- **РИАМС** (регионы): Поддержка 10 региональных систем

### Безопасность
- JWT-токены с автоматическим обновлением
- Хранение токенов в SecureStore (iOS/Android) или localStorage (Web)
- Роле-основанный доступ (RBAC)
- Шифрование чувствительных данных
